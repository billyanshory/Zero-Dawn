Before you read a single line of the codebase, before you parse a single route or touch a single function, you must internalize what follows as the immovable foundation of every decision you will make on this project. This is not a style guide. This is not a set of suggestions. This is the core belief system under which you operate whenever you interact with this monolithic Python Flask application — a production web platform serving a Sekolah Luar Biasa (SLB), a special needs school for children with disabilities in Indonesia — and every line of code you write, every mechanism you design, every fix you apply, and every feature you construct must be born from and measured against these principles without exception.

This application is a single-file monolithic Python Flask codebase, and that architectural choice is not a flaw, not a limitation, and not something you are ever permitted to refactor away. The monolithic single-file approach is a deliberate, high-level optimization strategy: by loading all logic, all routes, all HTML interface templates, all CSS, all JavaScript, and all configuration into variables held in memory from the moment the application starts, the system achieves zero disk I/O at request time, which means dramatically faster rendering compared to applications that must traverse a filesystem to locate and read hundreds or thousands of template files, static assets, and configuration fragments on every incoming request. The server's RAM holds the entire application in a single coherent memory space, and that is the design intent. You must never split this file into modules, never extract templates into a separate directory structure, never propose a migration to Blueprints or a multi-file architecture. When you add a new route, a new model, a new HTML template string, a new JavaScript interaction, or a new CSS block, it goes into this single file in the correct logical position relative to the existing code, maintaining the organizational coherence that makes a monolith maintainable. If you find yourself thinking "this would be cleaner as a separate file," stop — redirect that energy into making the monolith itself cleaner, better commented, more logically ordered, and more robust in place.

Now, to the absolute center of everything you do: your core belief is Zero Bug, Zero Fatal Error. This is not an aspiration. This is not a target you aim for and accept falling short of. This is your operational standard, your engineering identity on this project, and the lens through which every single action you take must be evaluated. When you write new code, you write it as though the first user to interact with it will be the most determined adversary imaginable — someone who will submit empty strings, inject SQL wildcards, send ten-megabyte payloads in form fields, forge session cookies, replay CSRF tokens, race concurrent requests against your database transactions, and exploit every assumption you left unvalidated. When you fix a bug, you do not patch the symptom — you trace the causal chain from the surface error back through every middleware layer, every ORM interaction, every session state mutation, every cached value, every implicit assumption, until you have found the true root cause, and then you seal that path permanently, structurally, and irreversibly, such that no future change to the codebase could accidentally reopen it. A fix that could regress is not a fix. A workaround that defers the problem is not acceptable. You resolve it completely or you do not claim it is resolved.

Every time you are about to commit a change, you must mentally execute a verification cycle that touches every critical dimension of the codebase. This is not optional. This is how you achieve zero bugs rather than merely intending to. The cycle begins with backend correctness: you verify that every route you have touched or created validates all input before it reaches business logic — checking types, enforcing minimum and maximum lengths, whitelisting allowed characters, rejecting unexpected fields, escaping SQL LIKE wildcards in search parameters, and stripping or escaping any HTML or markup before it reaches template rendering or PDF generation. You verify that every database interaction is wrapped in proper transaction handling — meaning every `db.session.commit()` is enclosed in a `try/except` block where the `except` clause calls `db.session.rollback()` before re-raising or returning an error response, because a failed commit without a rollback leaves the SQLAlchemy session in a poisoned state that will corrupt subsequent requests on the same database connection. You verify that every route which reads or writes sensitive data — student records, medical profiles, tantrum logs, emotion journals, nutrition tracking, IEP documents, burnout assessments, emergency contacts — checks not only the caller's role via `session.get('peran')` but also their ownership relationship to the data they are accessing, because a parent who is authenticated does not automatically have the right to view another parent's child's medical records, and a teacher who is authenticated does not automatically have the right to modify another teacher's class data. You verify that CSRF protection is active on every POST route without exception — if a route was previously exempted with `@csrf.exempt`, you must remove that exemption and ensure the frontend sends the CSRF token via the `X-CSRFToken` header in its `fetch()` or AJAX calls. You verify that the session is regenerated after successful authentication — calling `session.clear()` and then re-populating the session data after `check_password_hash` succeeds — to prevent session fixation attacks where an attacker who knows a pre-authentication session cookie can hijack the post-login session.

The cycle continues with frontend correctness: you verify that every form you have touched or created includes client-side validation — `required` attributes on mandatory fields, `maxlength` and `minlength` attributes on text inputs, `pattern` attributes where format constraints exist, and JavaScript validation that runs before the form submits to the server, so that obviously invalid input never wastes a round-trip. You verify that every form submission button is disabled after the first click and re-enabled only after the server response is received or an error timeout is reached, preventing double-submit scenarios that create duplicate database records, duplicate file uploads, or duplicate PDF generations. You verify that every `fetch()` call and every AJAX interaction provides visual feedback to the user — a loading spinner, a disabled button state, a progress indicator — so that the user knows their action was registered and the system is processing. You verify that every `fetch()` call has proper error handling in both its `.then()` chain and its `.catch()` block, and that network failures or server errors result in a user-visible error message rather than a silent failure that leaves the user wondering why nothing happened. You verify that search inputs are debounced — a minimum of 300 milliseconds — so that typing a student's name does not fire a database query on every single keystroke.

The cycle then moves to data integrity: you verify that every foreign key reference submitted by the client — `anak_id`, `siswa_id`, `guru_id`, or any other relational identifier — is validated against the database before you attempt to insert or update a record that depends on it, because accepting a nonexistent foreign key will result in a PostgreSQL `IntegrityError` that surfaces as an unhandled 500 error to the user rather than a meaningful "student not found" message. You verify that unique constraint violations — on fields like `nik`, `username`, or any other column with a uniqueness requirement — are caught explicitly with a `try/except IntegrityError` block that returns a human-readable error message rather than letting the exception propagate to the global error handler. You verify that no deletion in the system is a hard delete — every model that represents meaningful data should use a soft-delete pattern with an `is_deleted` boolean flag and a `deleted_at` timestamp, and all queries should filter out soft-deleted records by default, because in a school managing medical records and behavioral data for children with disabilities, accidental permanent deletion of a student's tantrum history or medical profile is catastrophic and irreversible. You verify that file uploads generate unique filenames using UUIDs or timestamp-plus-UUID combinations rather than content-based hashes, because two files with identical content uploaded simultaneously would generate the same hash-based filename and one would silently overwrite the other.

The cycle then addresses security at every layer: you verify that the `allow_unsafe_werkzeug=True` flag and `debug=True` setting in the application's startup block are either removed entirely or guarded behind a strict environment variable check that ensures they can never activate in production, because the Werkzeug debugger exposes a remote code execution console that would give any attacker complete control of the server. You verify that CORS configuration on Flask-SocketIO explicitly rejects wildcard origins — if the `ALLOWED_ORIGINS` environment variable is empty or set to `*`, the application must refuse to start or must fall back to a restrictive default rather than allowing every origin on the internet to open WebSocket connections to student data streams. You verify that the `send_from_directory` route serving uploaded files has defense-in-depth filename validation on the serve path, not just the upload path, rejecting any filename that contains path traversal sequences regardless of encoding. You verify that PDF generation using ReportLab escapes or strips all HTML from user-supplied strings before passing them to `Paragraph()` objects, because ReportLab interprets HTML markup and a crafted input like `<img src=x onerror=...>` or deeply nested tags can crash the renderer or produce malformed output. You verify that the application sends security headers on every response — `Content-Security-Policy` with a strict policy that accounts for the inline scripts and CDN resources the application uses, `X-Frame-Options: DENY` to prevent clickjacking, `X-Content-Type-Options: nosniff` to prevent MIME-sniffing attacks on uploaded files, and `Strict-Transport-Security` to enforce HTTPS. You verify that rate limiting is applied to at minimum the login route, the registration route, the brankas unlock route, and all API calculation endpoints, because without rate limiting these endpoints are vulnerable to brute-force attacks, credential stuffing, and denial-of-service through computational exhaustion.

The cycle then examines performance and optimization: you verify that every query which returns a list of records is paginated rather than using `.all()`, because as the school accumulates years of tantrum logs, emotion journals, portfolio entries, and nutrition records, unbounded queries will load thousands or tens of thousands of records into memory on every request, causing progressively slower responses and eventual memory exhaustion. You verify that frequently filtered and sorted columns — particularly `siswa_id`/`anak_id` and `created_at` on models like `TantrumLog`, `ReactionTimeLog`, `KognitifEmosiLog`, `KognitifBentukLog`, `EmotionJournal`, and `StudentPortfolio` — have composite database indexes that match the query patterns, specifically `(anak_id, created_at DESC)` for the most common access pattern of "show me this student's records in reverse chronological order." You verify that dashboard routes and any route that accesses a student along with their related records — medical profiles, emotion journals, tantrum logs — use eager loading via `joinedload()` or `selectinload()` rather than relying on SQLAlchemy's default lazy loading, which would issue a separate database query for every related record accessed in the template, creating N+1 query storms that multiply response times. You verify that heavy JavaScript libraries like Chart.js are included conditionally, only on pages that actually render charts, rather than being loaded universally in the base layout where pages like the sign language dictionary or emergency contact page pay the download cost for a library they never use. You verify that HTTP response compression is configured — either through Flask-Compress middleware or at the reverse proxy level — because the massive HTML template strings embedded in the monolith, some spanning thousands of lines of inline HTML, CSS, and JavaScript, are sent uncompressed without this, and compression typically reduces payload sizes by 70-80%. You verify that any external resources fetched at startup — such as emoji icon files from CDN servers — are either bundled as static assets to eliminate the runtime dependency or fetched asynchronously so that a slow or unreachable CDN does not block application startup.

The cycle then covers error handling and resilience: you verify that the global exception handler differentiates between HTTP error types — returning appropriate 400, 403, 404, and 500 responses with distinct error pages rather than a single generic catch-all — and that it does not catch `KeyboardInterrupt` or `SystemExit`, which would prevent clean process shutdown. You verify that every API route — any route consumed by frontend JavaScript via `fetch()` — returns JSON error responses on failure, not HTML, because the frontend code that parses the response expects JSON and will fail silently or throw an uncaught exception when it receives HTML from the generic error handler. You verify that the `seed_slb_data` function and any other initialization code logs errors through the application logger rather than `print()`, because `print()` output goes to stdout which is invisible in production logging configurations, meaning initialization failures would go completely undetected. You verify that scheduled background jobs — particularly the medication reminder checker — have proper error logging and alerting, because if a Redis connection drops and the medication reminder job starts failing silently, children with disabilities who depend on timed medication prompts will miss their doses without anyone knowing the system stopped working. You verify that long-running routes — especially PDF generation routes that may fetch external resources over the network — have timeout protection so that a slow network response cannot tie up a worker process indefinitely.

The cycle then examines WebSocket and concurrency correctness: you verify that SocketIO event handlers use room-based isolation keyed by `siswa_id` or another appropriate scope, so that when Teacher A is monitoring Student A's heart rate data and Teacher B is monitoring Student B, they each receive only the data relevant to their student rather than a broadcast of all data to all connected clients. You verify that the SocketIO instance has `ping_timeout` and `ping_interval` configured to detect and clean up dead WebSocket connections from closed browser tabs, because without this configuration, ghost connections accumulate in memory. You verify that the scheduler's distributed lock mechanism — the Redis-based `slb_scheduler_master` lock — has a TTL short enough that if the master worker crashes, another worker can acquire the lock within minutes rather than waiting up to an hour, and that the lock holder renews it periodically to prevent premature expiry during normal operation. You verify that push notification delivery is idempotent — tracking the last-sent timestamp per subscription per notification type — so that a scheduler restart or duplicate job execution does not result in duplicate notifications being sent to parents.

The cycle then addresses observability and monitoring: you verify that the application has a health check endpoint — a simple `/health` or `/ping` route that verifies the application process is running and the database connection is alive — because deployment platforms, load balancers, and monitoring systems need this endpoint to determine whether the application is healthy. You verify that logging includes not just errors but also informational events — login attempts (successful and failed), authorization denials, data modifications, and scheduled job executions — because without an audit trail, there is no way to investigate security incidents, debug intermittent issues, or satisfy data protection compliance requirements for a system handling children's medical and behavioral data. You verify that request-ID tracking is implemented — a middleware that generates or propagates a unique identifier for each request and includes it in every log line — so that when a user reports a problem, it can be correlated to specific log entries across the entire request lifecycle.

The cycle then examines deployment safety: you verify that `db.create_all()` is not the sole schema management mechanism, because while it creates missing tables, it cannot add columns to existing tables, change column types, drop constraints, or perform any schema evolution — meaning that any model change after initial deployment will fail silently, leaving the database schema out of sync with the application code. You verify that uploaded files are configured to be served by the reverse proxy (Nginx) rather than the Python application process, because every file download served by the application occupies a Gunicorn worker for the duration of the transfer, and a few simultaneous large file downloads can exhaust the worker pool and make the entire application unresponsive.

The cycle finally addresses domain-specific data safety for the SLB context: you verify that access to medical data — the `ProfilMedisSiswa` model containing allergy information, tantrum triggers, and emergency contacts for children with disabilities — is logged in an audit trail that records who accessed or modified the data and when, because Indonesian data protection law (UU PDP) may require access logging for minor health data, and because ethical handling of this extraordinarily sensitive information demands accountability. You verify that the emergency contact route requires authentication, because a stranger who picks up an unlocked device should not be able to view a child's emergency medical contacts and family information. You verify that IEP (Individual Education Plan) PDF generation requires role-based authorization, not just rate limiting, because without a role check anyone who can reach the route can generate a PDF document containing a student's name, class, and educational assessment data. You verify that behavioral data — emotion journals, tantrum logs, burnout assessments — is handled with awareness that this data describes real children with real disabilities and their families under real stress, and that any aggregation, export, or sharing mechanism you build must separate personally identifiable information from analytical data so that individual families cannot be identified from aggregate reports.

The final principle is about regression awareness, and it is perhaps the most operationally critical of all. Every time you apply a batch of changes — whether it is five bug fixes, three new features, or a combination — you must verify that your changes have not broken anything that was previously working. This means you must understand the existing codebase well enough to know what your changes might affect. If you modify an import statement, you must verify that the import ordering still satisfies the eventlet monkey-patching constraint — `eventlet.monkey_patch()` must execute before any `import redis` or any other network library import, because violating this ordering breaks greenlet safety and causes subtle, intermittent concurrency failures that are extraordinarily difficult to diagnose in production. If you modify a model, you must verify that every route, every query, and every template that references that model's fields still uses the correct field names, because a renamed field that is still referenced by its old name in a route handler will not fail at startup — it will fail at runtime when a user hits that specific route, producing a 500 error that may not be discovered for days or weeks. If you modify a template string, you must verify that any CSS classes, JavaScript function references, and HTML element IDs that the template depends on still exist and still work correctly — a missing Tailwind CSS import, a renamed JavaScript function, or a changed element ID can silently destroy an entire user interface. If you add a new dependency, you must verify that the import succeeds gracefully — wrapping the import in a `try/except ImportError` with a clear log message and a sensible fallback, so that a missing package does not crash the entire application at startup. You must treat every change as potentially introducing a regression until you have specifically verified that it has not, and you must never assume that a change is safe simply because it looks simple. The most catastrophic regressions in this codebase's history have come from changes that appeared trivial — a reordered import that broke eventlet, a missing CSS file reference that destroyed every page's layout, a renamed model field that crashed a critical route. Vigilance against regression is not paranoia; it is engineering discipline.
